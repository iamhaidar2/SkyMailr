Quick Start

This Quick Start guide for SkyMailr walks through what you need to connect an app and send mail. Start with Connect your app end-to-end below, then use the tabs: Coding Agent for a single prompt to paste into an assistant, or You Code for the same integration with more detail and copy-paste examples.

Connect your app end-to-end

Work from top to bottom. Steps 1–6 happen in SkyMailr; steps 7–8 connect your own product and where it runs.

  1. Create a connected app (your product or environment) under Connected apps.
  2. Add an API key on API keys: choose the app, generate a key, and copy the secret once to a password manager—you will not see it again.
  3. If mail should come from your own domain (for example noreply@yourdomain.com): open Sending domains, add the domain, add the DNS records your DNS host shows, then run Check records in SkyMailr until it shows verified.
  4. Set the From name and address under Sender profiles for that connected app.
  5. Create email templates under Email templates and have an owner or admin approve them for sending.
  6. Optional: Workflows for automated sequences and Members so teammates can help.
  7. Add SkyMailr to your app’s code. Your product should read the secret from configuration (for example SKYMAILR_API_KEY), never from source control. When you send mail, call SkyMailr’s send-template API with that secret in the Authorization header. Use the Coding Agent or You Code tab below for prompts and examples, or follow docs/06-api-and-auth.md in the SkyMailr repo.
  8. Configure your app host (for example Railway, Heroku, or Fly.io). In the host’s Variables or Environment screen, add SKYMAILR_BASE_URL = {{ skymailr_site_root }} (your SkyMailr site address only—no /api/v1 at the end) and SKYMAILR_API_KEY = the secret from step 2. Save, then redeploy so the server restarts and picks up the values.

If your app only reads environment variables when it starts, you must redeploy after every change to those variables. The Dashboard setup checklist mirrors these steps in a shorter list.

You are helping integrate an application with SkyMailr (hosted email sending).

SkyMailr REST API base (all paths below this URL): {{ api_base_url }}

Authentication
Send this HTTP header on API requests:
  Authorization: Bearer <SKYMAILR_API_KEY>
The human creates the secret key once in the SkyMailr web app under API keys, then stores it only in the app’s environment or secret manager (never in git or the browser).

Customer portal (where humans configure SkyMailr): {{ portal_app_root }}

Main endpoint to send a templated email
  POST {{ api_base_url }}/messages/send-template/
  Content-Type: application/json

Request body (shape):
{
  "template_key": "string matching an approved template in SkyMailr",
  "to_email": "recipient@example.com",
  "context": { "key": "values used inside the template" },
  "source_app": "stable short id for this product (for logs)",
  "message_type": "transactional",
  "idempotency_key": "optional unique string so retries do not duplicate sends"
}

Before sends succeed, the human must complete in the portal:
- At least one connected app (their product or environment).
- An API key for that app, copied to secure storage.
- If mail must come From their own domain: add DNS records SkyMailr shows, then verify under Sending domains.
- A sender profile (the visible From name and address) for that connected app.
- An email template saved and approved; template_key in JSON must match that template.

You Code — integrate SkyMailr yourself

This is the same integration as the Coding Agent tab, with more step-by-step detail, explicit environment variable names, and small examples you can copy.

Before you call the API

Complete Connect your app end-to-end above (steps 1–6): connected app, API key, optional sending domain + DNS, sender profile, approved templates. Without those, requests return errors or mail will not send as you expect.

Environment variables (your app + host)

  • SKYMAILR_BASE_URL — your SkyMailr site root only, e.g. {{ skymailr_site_root }}. Do not append /api/v1; your code or HTTP client adds path /api/v1/messages/send-template/.
  • SKYMAILR_API_KEY — the raw secret from API keys. Set it in Railway / Heroku / Fly “Variables”, not in git.

HTTP request

  • Method & URL: POST {{ api_base_url }}/messages/send-template/
  • Headers: Authorization: Bearer <SKYMAILR_API_KEY> and Content-Type: application/json
  • JSON body fields: template_key (must match an approved template), to_email, context (object of template variables), source_app (stable id for logs), message_type (e.g. transactional), optional idempotency_key for safe retries.

Examples

Replace template key, recipient, and context keys with your own. Each block has a Copy button.

Python (httpx)

import os
import httpx

base = os.environ["SKYMAILR_BASE_URL"].rstrip("/")
url = f"{base}/api/v1/messages/send-template/"
headers = {
    "Authorization": f"Bearer {os.environ['SKYMAILR_API_KEY']}",
    "Content-Type": "application/json",
}
body = {
    "template_key": "welcome_new_user",
    "to_email": "user@example.com",
    "context": {"user_name": "Ada"},
    "source_app": "my_product",
    "message_type": "transactional",
}
r = httpx.post(url, json=body, headers=headers, timeout=30.0)
r.raise_for_status()
print(r.json())

JavaScript (Node / fetch)

const base = process.env.SKYMAILR_BASE_URL.replace(/\/$/, "");
const url = `${base}/api/v1/messages/send-template/`;
const res = await fetch(url, {
  method: "POST",
  headers: {
    Authorization: `Bearer ${process.env.SKYMAILR_API_KEY}`,
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    template_key: "welcome_new_user",
    to_email: "user@example.com",
    context: { user_name: "Ada" },
    source_app: "my_product",
    message_type: "transactional",
  }),
});
if (!res.ok) throw new Error(await res.text());
console.log(await res.json());

cURL

curl -sS -X POST "{{ api_base_url }}/messages/send-template/" \
  -H "Authorization: Bearer $SKYMAILR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"template_key":"welcome_new_user","to_email":"user@example.com","context":{"user_name":"Ada"},"source_app":"cli","message_type":"transactional"}'
