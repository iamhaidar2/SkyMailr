Webhooks & delivery events

See what happened to your mail after send: delivered, bounced, opened, and more. Use this page to debug a recipient or campaign; use the API to integrate with your own systems.

Why, when, and how

Inbound webhooks (provider → SkyMailr): Your mail provider (e.g. Postal) sends delivery feedback to SkyMailr so we can update message status, record bounces/complaints, and apply suppressions. You do not call this URL from your product code—your provider is configured once on the server. The path pattern is:

{{ inbound_webhook_url_pattern }}

Replace <provider_name> with the provider configured in SkyMailr (e.g. postal). Operators configure the exact URL in the provider / Postal project settings.

Outbound events (SkyMailr → your app): Poll or stream events from your backend using the API—there is a stable events feed per message:

GET {{ request.scheme }}://{{ request.get_host }}/api/v1/messages/<message_uuid>/events/
Authorization: Bearer <your API key>

Create an API key under API keys. The response lists events (delivered, bounced, etc.) with payloads. The table below is the same data, filtered to your account.

Webhook secret (per connected app): On each app’s Manage app page you can set an optional webhook secret for future signed callbacks or integrations. Keep it in a secret manager; if you rotate it, update any code that verifies signatures.
